PART 1 · WHAT A SKILL IS

A Skill Is a Reusable Workflow You Write Once and Invoke Forever

  • A skill is a markdown file that gives Claude a named, repeatable procedure.
  • You keep pasting the same instructions? That belongs in a skill.
  • Unlike CLAUDE.md, the skill body only loads when you call it, saving context budget.
  • You can invoke it yourself (/skill-name) or let Claude recognize the pattern and load it.
  • No coding required. If you can write a checklist, you can write a skill.
A skill is your standard operating procedure, stored where Claude can find and run it.

PART 2 · FOUR PRIMITIVES

Four Primitives in Claude Code - Know Which to Reach For

SKILL
Reusable workflow
A markdown procedure Claude loads and follows. You or Claude can invoke it. Persistent across sessions.
SLASH COMMAND
One-off shortcut
A shortcut typed with / to invoke a skill or a built-in. Skills are the more powerful successor for anything with real structure.
SUBAGENT
Isolated specialist
Runs in its own context window with its own tools. Keeps noisy exploration out of your main thread. Returns only a summary.
MCP SERVER
External tool bridge
Connects Claude to live systems: databases, Jira, Slack, GitHub. Skills tell Claude how to think. MCP gives Claude hands to act on outside data.
Start with a skill. Add a subagent when you need isolation. Add MCP when you need live data.

PART 3 · ANATOMY OF SKILL.md
LIVE DEMO

A SKILL.md Has Two Parts: Frontmatter and Body

  • YAML frontmatter (the --- block) tells Claude when and how to use the skill.
  • name: the display name and the /command you type. Optional - defaults to folder name.
  • description: the trigger. Claude reads this to decide whether to load the skill.
  • The markdown body is the instructions Claude follows when the skill runs.
  • Only description is recommended. All other frontmatter fields are optional.
---
name: standup-summary
description: |
  Summarize daily standup notes
  into a structured digest.
  Trigger on 'summarize the
  standup' or standup-*.md files.
---

# Standup Summary

Turn raw standup notes into a
structured markdown summary.

## Output template
...
YAML (pronounced 'yam-ul') is just structured text between --- markers. No code needed.

PART 4 · THE TRIGGER MECHANISM

Skills Load Only When Their Description Matches

Skill descriptions are always in Claude's context. The full body is not.
ALWAYS IN CONTEXT
description field (truncated at 1,536 characters)
Claude scans this every turn to decide whether to load the skill.
LOADS ON TRIGGER
full SKILL.md body + supporting files
Enters conversation when Claude matches the description or you type /skill-name.
LOADS ON DEMAND
references/, scripts/, assets/ subdirectories
Claude reads these only when the skill body explicitly points to them.
Put trigger phrases like 'summarize the standup' or 'draft an email' directly in description. That is the whole mechanism.

PART 5 · PROGRESSIVE DISCLOSURE

Three Layers Load in Order, Not All at Once

Progressive disclosure means Claude pays only for what it actually needs.
1
Metadata layer
Name + description. Always in context, zero extra cost. Claude uses this to decide.
2
Body layer
SKILL.md instructions. Loads the moment the skill triggers. Stays for the rest of the session.
3
Resource layer
references/, scripts/, assets/ files. Claude reads these only when the body explicitly points to them.
Keep the description sharp, keep the body focused, park heavy reference docs in subfolders.

PART 5 · WHERE SKILLS LIVE

Personal Skills Follow You. Project Skills Stay in the Repo.

PERSONAL
~/.claude/skills/<skill-name>/
  • Available in every project you open.
  • Only you can see and use them.
  • Best for: email drafting, standup summary, presentation builder, OKR writer.
  • Create once. Reuse across every workspace.
PROJECT
.claude/skills/<skill-name>/
  • Lives inside the repo. Shared with the team.
  • Committed to version control alongside code.
  • Best for: team-specific workflows, shared templates, project conventions.
  • Check it in once. The whole team benefits.
Your personal skill folder is your private toolkit. The project folder is the team playbook.

PART 5 · FOLDER STRUCTURE

One Folder Per Skill. SKILL.md Under 500 Lines.

  • Each skill gets its own folder. The folder name becomes the /command.
  • SKILL.md is required. All other files are optional but powerful.
  • references/ holds detailed docs Claude loads on demand, not on startup.
  • scripts/ holds code Claude can execute. assets/ holds templates, examples.
  • Target 100 to 200 lines, hard cap 500. Overflow goes to subfolders.
my-skill/
  SKILL.md          <- required
  references/
    api-docs.md     <- load on demand
    style-guide.md  <- load on demand
  scripts/
    validate.sh     <- executed
  assets/
    template.md     <- examples
    sample.md       <- examples
If your SKILL.md is approaching 500 lines, move reference docs to references/ and link to them.

PART 5 · WHEN TO CREATE A SKILL

The 3-Repeat Rule: Paste It Three Times, Then Make It a Skill

SKIP THE SKILL
  • You did it once and won't repeat it.
  • The task changes so much each time that no template fits.
  • You're still figuring out the right format.
  • It's a one-liner prompt with no real structure.
MAKE THE SKILL
  • You paste the same instructions 3 or more times.
  • A section of CLAUDE.md has grown into a procedure.
  • A team member asks 'how do I get Claude to do X?'
  • The output format is stable and reusable.
If the task is still evolving, keep it in CLAUDE.md. Once stable and repeated, graduate it to a skill.

PART 5 · INVOCATION

Two Ways to Invoke a Skill. Claude Decides or You Decide.

1
Auto-trigger
You say 'summarize the standup'. Claude scans descriptions, finds a match, loads the skill. No /command needed. Best for: reference skills, knowledge skills, anything that should run silently when relevant.
2
Direct invocation
You type /standup-summary. Claude loads the skill immediately, regardless of description match. Best for: workflow skills with side effects (send, commit, deploy) where you control timing.
Tip: use disable-model-invocation: true in frontmatter to block auto-trigger. Claude can then only be called by you.
If the skill does something consequential (sends an email, updates a doc), force manual invocation.

PART 6 · REAL SKILL: EMAIL-DRAFTER
LIVE DEMO

email-drafter: Say 'Draft an Email'. Get a Structured Draft.

  • Trigger: 'draft an email', 'write an email to', 'follow up with', 'reply to this thread'.
  • Framework inside: Barbara Minto's Pyramid Principle (BLUF, SCQA, MECE).
  • Output: Subject line + body. No preamble, no filler, ends on a clear next step.
  • The skill classifies intent first: executive update, decision request, cross-team ask.
  • Trigger also fires on indirect signals: 'tell my VP the launch is slipping'.
# Output example

**Subject:** Decision needed
by Wed: ship Smart Follow-Up
April 24 or slip 2 weeks

QA found a blocker in the
confidence model Friday.
Recommending a slip
from Apr 24 to May 8.

Two options:
1. Ship Apr 24, 8% accuracy drop.
2. Slip to May 8, fix the model.

Can you confirm by Wednesday?
BLUF means 'Bottom Line Up Front'. Minto's Pyramid means the most important point comes first, always.

PART 6 · REAL SKILL: STANDUP-SUMMARY
LIVE DEMO

standup-summary: Point at a File. Get a 60-Second Digest.

  • Trigger: 'summarize the standup', 'what happened in standup', or any daily-standup-*.md file.
  • What Claude does: reads the file end to end, extracts progress, blockers, decisions, actions.
  • Output: structured markdown saved to outputs/standup-summary-DD-MM-YYYY.md.
  • Every action item gets an owner AND a due date. Both are required by the skill instructions.
  • The skill converts all dates to DD-MM-YYYY, even if the notes use other formats.
# Output structure

## TL;DR
- 3 bullets a skip-level
  needs to know

## Progress by person
| Person | Yesterday | Today |
|--------|-----------|-------|
| Remi   | ... | ... |

## Blockers and risks
- corpus noise, owner: Remi

## Action items
- [ ] spot check corpus
  owner: Remi, due: 20-03-2026
This skill turns messy Slack-style standup notes into a digest your VP could read in 60 seconds.

PART 7 · WHEN NOT TO MAKE A SKILL

Not Every Repeated Task Earns a Skill

SKIP THE SKILL
One-off tasks
You ran it once. You won't run it again. Don't build the abstraction.
BETTER ALTERNATIVE
Just paste the instructions directly. Save the 20 minutes.
SKIP THE SKILL
Hyper-specific to one meeting
The template only fits one context. It will confuse Claude elsewhere.
BETTER ALTERNATIVE
Add a one-off section to CLAUDE.md instead. Archive after the meeting.
SKIP THE SKILL
Workflow still evolving
Skill bodies stay in context after load. Instability costs tokens.
BETTER ALTERNATIVE
Keep it in CLAUDE.md until the format is stable, then graduate it.
SKIP THE SKILL
A fact, not a procedure
Single facts, rules, and preferences belong in CLAUDE.md, not a skill.
BETTER ALTERNATIVE
CLAUDE.md is always in context. Skills load on demand. Facts should always load.
Skills are for stable, repeated workflows. Facts belong in CLAUDE.md. One-offs go directly in chat.

PART 7 · BUILD YOUR FIRST SKILL

Five Steps from Zero to a Working Skill

1
Identify the pattern.
Pick a task you paste instructions for 3 or more times. That is your candidate.
2
Create the folder.
mkdir -p ~/.claude/skills/my-skill/ for personal, or .claude/skills/my-skill/ for project.
3
Write SKILL.md.
Frontmatter first: name, description with trigger phrases. Then instructions in markdown.
4
Test auto-trigger.
In Antigravity IDE, say the trigger phrase naturally. Watch whether Claude loads the skill.
5
Iterate the description.
If it does not trigger, add more trigger phrases. If it triggers too often, make it more specific.
Step 5 is the most important one. A skill that never triggers is just a markdown file nobody reads.

PART 7 · WHERE DOES IT BELONG?

Four Containers, One Question Each

Is it a project fact, a standing rule, a repeatable procedure, or a one-off? Each has a home.
PROJECT FACT
CLAUDE.md
What this project is: stack, file map, conventions, hard constraints for this repo. Always loaded.
STANDING RULE
Memory
A preference Claude applies everywhere: date format, tone, naming. Add with #, view with /memory. Split out of CLAUDE.md by design.
STABLE PROCEDURE
Make it a skill
Loads on demand when the description matches. Examples: standup summary, email drafter, PRD writer, OKR writer.
ONE-OFF TASK
Paste it in chat
No overhead, no file. Examples: analyzing a specific doc once, a question about one meeting.
A recurring rule is the trap: it belongs in memory, not CLAUDE.md and not a skill.

WRAP-UP · FIVE THINGS TO REMEMBER

The Five Things That Actually Matter

  • A skill is a markdown file with YAML frontmatter and an instructions body.
  • The description field is the trigger. Load trigger phrases there, not in the body.
  • Progressive disclosure: description always, body on trigger, resources on demand.
  • Three times pasting the same thing = time to make a skill.
  • Project facts in CLAUDE.md. Standing rules in memory. Procedures in skills. One-offs in chat.
A skill is leverage. Build it once, use it on every project, share it with the team.

REFERENCE · FAQ

Five Questions You'll Be Asked Back at Work

1
Skill vs Subagent vs MCP vs Plugin - what is the difference?
Per docs: a Skill is a SKILL.md file Claude loads when its description matches. A Subagent runs in its own context window. An MCP server is a connector to external tools. Plugins package skills with other extensions.
2
Will Claude Code suggest skills, or do I have to know what to create?
Claude does not auto-suggest. Docs recommend creating one when you keep pasting the same instructions or a CLAUDE.md section grows into a procedure. Use the bundled skill-creator to scaffold.
3
If I add a skill in this project, will it work in my other projects?
No. Project skills live in .claude/skills/ and load only in that repo. Personal skills live in ~/.claude/skills/ and apply across all your projects. Enterprise overrides both.
4
How do I know if my skill description is good enough to trigger correctly?
Docs: include keywords users would naturally say; put the key use case first. The description is capped at 1,536 chars in the listing. Verify with 'What skills are available?' inside Claude.
5
What about Cursor rules - aren't those the same as skills?
Different mechanic. Skills load only when their description matches and enter context only on invocation. Always-on rules consume context every turn. Skills follow the Agent Skills open standard.
