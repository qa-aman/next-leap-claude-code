Claude Code for PMs - Hands-on Lab

Build Five Agents by Describing Them to Claude

A subagent is one Markdown file with a bit of YAML on top. You do not hand-write it: you tell Claude what you want, it writes the file into .claude/agents/. This handout walks you through nine subagents across the five classic workflow patterns.

Reference: Anthropic's official subagents guide. Pattern set: Anthropic, "Building Effective Agents." All file paths refer to this MeetFlow workshop repo.

The /agents wizard is gone. As of Claude Code v2.1.198, typing /agents no longer opens the interactive creation wizard (the old Running and Library tabs). It just prints a reminder to ask Claude or edit .claude/agents/ directly. Everything else is unchanged: the file format, the frontmatter fields, and the .claude/agents/ (project) and ~/.claude/agents/ (personal) locations all still work. This handout uses the current flow.
You are building these from a clean slate. For this lab the eight prebuilt agents have been cleared out of .claude/agents/, so you create each one fresh (the ninth, competitor-deep-dive, is new anyway). If your copy of the repo still has any of them, creating an agent with the same name just overwrites it, which is fine here.
Build order matters in two places. Create the three Routing specialists (prd-drafter, competitor-snapshot, churn-diagnoser) before the pm-request-router, and create prd-critic before prd-quality-loop. A dispatcher that points at an agent you have not built yet fails at runtime. Each section below flags this where it applies.

What you will build

  1. Prompt Chaining → interview-insight-synthesizer
  2. Routing → pm-request-router (+ 3 specialists)
  3. Parallelization → feedback-triangulator
  4. Orchestrator-Workers → competitor-deep-dive
  5. Evaluator-Optimizer → prd-quality-loop (+ critic)

How to create a subagent now

There are two ways. Use the first for every agent in this handout. The second is what the first one produces, so it is worth seeing once.

Method A Ask Claude (recommended)

In the Claude Code chat, describe the agent and where to save it. Claude writes the file, fills in the frontmatter, and writes the system prompt for you.

you, in chat 
Create a project subagent in .claude/agents/ called "code-improver" that
scans files and suggests improvements for readability, performance, and
best practices. Make it read-only and have it use Sonnet.
Method B Write the file yourself

A subagent is just a Markdown file: YAML frontmatter on top, the system prompt as the body. Drop it in .claude/agents/.

.claude/agents/code-improver.md 
---
name: code-improver
description: Scans files and suggests improvements for readability,
  performance, and best practices. Use after writing or modifying code.
tools: Read, Grep, Glob
model: sonnet
---

You are a code improvement specialist. For each issue you find, explain
the problem, show the current code, and provide an improved version.

Only name and description are required. The rest have sensible defaults.

No restart needed, usually. Claude Code watches both agent folders and picks up new or edited files within a few seconds. PATTERN 01

Prompt Chaining

Decompose a task into sequential stages. Each stage has a quality gate before moving on.

The agent: interview-insight-synthesizer

Reads every transcript in 07-user-interviews/ and produces a ranked synthesis brief. Chains four stages: extract pains → cluster into themes → rank by frequency → write the brief. Single agent, no fan-out.

Method A Ask Claude - paste this in chat
Create a project subagent in .claude/agents/ called
"interview-insight-synthesizer" for MeetFlow, using Sonnet, color blue.
Give it Read, Glob, and Write only.

It reads every transcript in 07-user-interviews/ and produces a ranked
synthesis brief in FOUR sequential stages with a quality gate between each:

STAGE 1 - Extract raw pains: use Glob to list all files in
07-user-interviews/, read each transcript, extract every distinct user
pain, frustration, or unmet need. Gate: if fewer than 3 pains per
interview, re-read that interview.

STAGE 2 - Cluster into themes: group all pains into 4-7 themes by
semantic similarity. Gate: no theme may have only 1 pain (merge singletons).

STAGE 3 - Rank by frequency and severity: count how many distinct
interviews mention each theme. Tag severity as workflow-breaker /
annoyance / nice-to-fix. Produce a ranked table.

STAGE 4 - Write the brief: save a 1-page Markdown synthesis to
outputs/interview-synthesis.md (no date in the filename; overwrite if it
exists). Structure: TL;DR (3 bullets) -> top 3 themes (each with quote +
count) -> recommendations (1 line each).

Rules: cite the interview filename for every quote. Never invent quotes.

Trigger it when the user asks to synthesize user interviews, find common
pain points, or build a research brief from interview transcripts.
Method B Or write the file
.claude/agents/interview-insight-synthesizer.md 
---
name: interview-insight-synthesizer
description: Synthesizes user interviews into a ranked research brief.
  Use when asked to find common pain points or build a research brief
  from the transcripts in 07-user-interviews/.
tools: Read, Glob, Write
model: sonnet
color: blue
---

You synthesize MeetFlow user interviews into a ranked brief. Work in four
sequential stages, with a quality gate before moving on:

1. EXTRACT - Glob 07-user-interviews/, read each transcript, list every
   distinct pain. Gate: fewer than 3 pains in an interview means re-read it.
2. CLUSTER - group pains into 4-7 themes by similarity. Gate: merge any
   single-pain theme into a larger one.
3. RANK - count distinct interviews per theme; tag severity as
   workflow-breaker / annoyance / nice-to-fix. Build a ranked table.
4. WRITE - save outputs/interview-synthesis.md (overwrite, no date):
   TL;DR (3 bullets), top 3 themes (quote + count each), recommendations.

Cite the interview filename for every quote. Never invent quotes.

Why these tools

Read and Glob to walk the transcripts, Write to save the brief. No Agent tool: this pattern is a single agent working through stages, not a dispatcher.

Test it

In chat: "Synthesize the user interviews and tell me the top 3 frustrations." Claude should delegate to this agent. Watch it move through all four stages and write the file to outputs/.

PATTERN 02

Routing

One agent classifies the request, then dispatches to a specialist. You build four agents: three specialists, then the router that calls them.

Build order matters. Create the three specialists (2A, 2B, 2C) before the router (2D). The router dispatches to specialists by name. If they do not exist yet, it fails at runtime.

Agent 2A: prd-drafter (specialist 1)

Method A Ask Claude - paste this
Create a project subagent in .claude/agents/ called "prd-drafter" for
MeetFlow, Sonnet, color green, with Read, Glob, Write.

It writes a concise one-page PRD. Before writing it reads
03-product-knowledge/company.md and 04-strategy/product-vision.md.

Output (Markdown, saved to outputs/prd-{feature-name}.md, no date):
1. Problem (3 lines, one stat from 06-user-feedback/ or 07-user-interviews/)
2. Target user (which persona from 05-user-personas/)
3. Solution (5 bullets, no implementation detail)
4. Success metric (1 metric, current baseline + target)
5. Risks (top 3, concrete)
6. Out of scope (top 3)

Trigger it when the user asks for a feature spec, PRD, one-pager, or
product requirements doc.

Agent 2D: pm-request-router (the router)

Method A Ask Claude - paste this
Create a project subagent in .claude/agents/ called "pm-request-router" for
MeetFlow, Sonnet, color orange. Give it the Agent tool only.

Its ONLY job is classification and dispatch. It must NEVER answer the
request itself.

Workflow:
1. Read the user's request.
2. Classify into exactly one of: PRD, COMPETITOR, CHURN, UNCLEAR.
3. Say it out loud: "Classified as: X. Dispatching Y."
4. Dispatch the matching specialist via the Agent tool:
   PRD -> prd-drafter, COMPETITOR -> competitor-snapshot,
   CHURN -> churn-diagnoser.
5. Pass the user's original request verbatim to the specialist.
6. If UNCLEAR, ask ONE clarifying question. Never guess.

Trigger it FIRST for any open-ended PM request, or when the user says
"help me with..." without specifying what they need.
Method B Or write the file
.claude/agents/pm-request-router.md 
---
name: pm-request-router
description: Triage agent for open-ended MeetFlow PM requests. Use first
  when the user says "help me with..." without naming a deliverable.
tools: Agent
model: sonnet
color: orange
---

Your ONLY job is classification and dispatch. Never answer the request
yourself. Steps: read the request, classify as PRD / COMPETITOR / CHURN /
UNCLEAR, announce "Classified as: X. Dispatching Y.", then dispatch via the
Agent tool (PRD to prd-drafter, COMPETITOR to competitor-snapshot, CHURN to
churn-diagnoser), passing the request verbatim. If UNCLEAR, ask exactly one
clarifying question. Never guess.

Why these tools

The router needs only the Agent tool. That is what lets one subagent dispatch to another (a subagent can spawn its own subagents). Without it, the router can classify but cannot hand off. It reads no files and writes nothing: it classifies the request in front of it and passes it on, and the specialists do all the reading and writing.

Test the routing

Try three requests and watch which specialist gets dispatched:

  • "Help me write a spec for Smart Follow-Up." → PRD (green agent runs)
  • "How does Granola compare to us?" → COMPETITOR (yellow agent runs)
  • "Why are Pro users leaving?" → CHURN (red agent runs)
PATTERN 03

Parallelization

One agent fires three workers at once, then merges their outputs.

The agent: feedback-triangulator

Dispatches three workers in parallel - interviews, Q1 survey, churn data - then surfaces issues that appear in MORE than one source.

Parallelization vs Orchestrator-Workers (do not confuse them). Parallelization uses a FIXED set of workers: always the same three sources, no matter the request. Orchestrator-Workers (Pattern 04) uses a DYNAMIC set: the agent picks which workers to spawn based on the input. Same parallel mechanics, different decision-making.
Method A Ask Claude - paste this
Create a project subagent in .claude/agents/ called "feedback-triangulator"
for MeetFlow, Sonnet, color purple, with Agent, Read, Glob, Write.

It triangulates user feedback from three independent sources in parallel.

1. Dispatch THREE workers IN PARALLEL via the Agent tool, all in a SINGLE
   message (do not wait between them). Use the general-purpose subagent type.
   Worker A: "Read every file in 07-user-interviews/. Return the top 5 user
   issues: title, 1-line description, source filename. Read nothing else."
   Worker B: "Read 06-user-feedback/. Return the top 5 issues from the Q1
   2026 survey: title, % of respondents, exact phrase. Read nothing else."
   Worker C: "Read 03-product-knowledge/company.md and any churn-related
   files. Return the top 5 issues tied to Pro churn (4.1% monthly): title,
   evidence, source file. Read nothing else."

2. When all three return, find issues that appear in 2 or 3 sources. Rank them.

3. Save outputs/triangulated-feedback.md (no date), two sections:
   Triangulated issues (2+ sources) = top priorities.
   Source-specific issues (1 source) = needs more investigation.

Always cite source files.

Trigger it when the user wants a triangulated view of feedback across
multiple sources.
Method B Or write the file
.claude/agents/feedback-triangulator.md 
---
name: feedback-triangulator
description: Cross-references interviews, survey, and churn data to find
  issues confirmed by 2+ sources. Use when the user wants a triangulated
  view of MeetFlow feedback.
tools: Agent, Read, Glob, Write
model: sonnet
color: purple
---

Dispatch THREE workers in a SINGLE message (general-purpose type) so they
run in parallel: A reads 07-user-interviews/, B reads 06-user-feedback/,
C reads 03-product-knowledge/company.md plus churn files. Each returns its
top 5 issues with source citations and reads nothing else. When all return,
rank issues by how many sources mention them. Save
outputs/triangulated-feedback.md (no date): triangulated issues (2+ sources)
first, then source-specific issues. Always cite source files.

Why these tools

The Agent tool is what enables the parallel dispatch. Read and Glob let the merge step re-check sources; Write saves the final brief.

Test it

"Triangulate the feedback. What is the top issue across sources?" In the subagent panel you will see three workers running at once. That is the parallel dispatch working.

PATTERN 04

Orchestrator-Workers

The orchestrator decides which workers to spawn based on the input. Different inputs produce different worker sets.

Unlike Routing (Pattern 02), which picks ONE specialist from a fixed menu, the orchestrator picks N workers, and the worker list changes per input. Unlike Parallelization (Pattern 03), which always dispatches the same fixed workers, the orchestrator decides the worker set at runtime.

The agent: competitor-deep-dive

Given any competitor name, this agent picks 3-5 research angles that matter for THAT competitor, then spawns a worker per angle.

Heads up: the repo already ships competitor-intel-analyst, a single-agent researcher with no worker fan-out. The one you build here, competitor-deep-dive, is the multi-worker orchestrator version. The names differ on purpose, so building it adds a new file next to the existing ones in .claude/agents/.
Method A Ask Claude - paste this
Create a project subagent in .claude/agents/ called "competitor-deep-dive"
for MeetFlow, Sonnet, color cyan, with Agent, WebFetch, Read, Glob, Write.

It runs deep competitive analysis on a named competitor.

1. Read 03-product-knowledge/competitive.md for what we already know.
2. Decide which research angles matter for THIS competitor. Available angles:
   Pricing & packaging, Feature surface, User reviews (G2/Reddit/X), Recent
   launches (last 90 days), Funding & team size, Positioning & ICP,
   Integration ecosystem.
3. Pick 3-5 angles. Justify the pick OUT LOUD before dispatching. Example:
   "For Granola, prioritizing Feature Surface, User Reviews, and Positioning
   - they are a prosumer Mac-native play, so pricing matters less than design."
4. Dispatch one worker per chosen angle IN PARALLEL via the Agent tool
   (general-purpose type). Each worker gets one angle only.
5. Synthesize into outputs/competitor-deepdive-{name}.md (no date):
   TL;DR -> Where they beat us -> Where we beat them -> Strategic
   implications (cite 04-strategy/product-vision.md).

Rules: ALWAYS justify angle selection before dispatch. Workers run in
parallel (one message). Cite worker outputs in the final brief.

Trigger it when the user wants a deep analysis of a specific competitor
by name.
Method B Or write the file
.claude/agents/competitor-deep-dive.md 
---
name: competitor-deep-dive
description: Multi-worker deep dive on a named competitor. Picks research
  angles per competitor, then fans out workers. Use for a deep, multi-angle
  analysis of one competitor by name.
tools: Agent, WebFetch, Read, Glob, Write
model: sonnet
color: cyan
---

Read 03-product-knowledge/competitive.md. Then pick 3-5 research angles that
matter for THIS competitor from: pricing, feature surface, user reviews,
recent launches, funding/team, positioning/ICP, integrations. Justify the
pick out loud BEFORE dispatching. Fan out one worker per angle in a single
message (general-purpose type), one angle each. Synthesize into
outputs/competitor-deepdive-{name}.md (no date): TL;DR, where they beat us,
where we beat them, strategic implications (cite 04-strategy/product-vision.md).
Always cite worker outputs.

Why these tools

Same as parallelization plus WebFetch, so workers can pull from competitor sites when the repo does not have the answer.

Test it twice

  • "Deep dive on Granola."
  • "Deep dive on Fireflies."

Verify the orchestrator picks different angles for each. If it picks the same angles for both, the angle-selection rule is too loose. Edit the agent file (or ask Claude to) and tighten it.

PATTERN 05

Evaluator-Optimizer

Two agents in a feedback loop. One writes. One critiques. The writer revises until the critic is satisfied.

Build order matters. Build 5A (the critic) first. The writer (5B) dispatches to the critic by name. If the critic does not exist yet, the loop fails.

Agent 5A: prd-critic (the critic)

Method A Ask Claude - paste this
Create a project subagent in .claude/agents/ called "prd-critic" for
MeetFlow, Sonnet, color pink, with the Read tool only.

It scores draft PRDs against a strict 5-dimension rubric. Persona: a
skeptical senior engineering manager. Be specific and brief.

Rubric (score each 0-2, total out of 10):
1. Problem clarity - real stat from real users? (0 none, 2 number + source)
2. User specificity - a named persona, not "users"? (0 generic, 2 named)
3. Solution scope - 5 bullets, no implementation, no creep? (0 jumbled, 2 tight)
4. Measurability - metric with current baseline + target? (0 vague, 2 both)
5. Risks - top 3 concrete and prioritized? (0 generic, 2 specific)

Output: "Score: X/10", then one line per dimension with a reason. If the
score is under 8, add "Top fixes" with 3 specific changes, each naming the
exact section and exact fix.

Rules: NEVER give a 10 - there is always something. Fixes must be specific,
e.g. "add the 34% action item accuracy stat from
04-strategy/product-vision.md," not "make the problem clearer."

Note it is used only when dispatched by prd-quality-loop.
Method B Or write the file
.claude/agents/prd-critic.md 
---
name: prd-critic
description: Scores a draft MeetFlow PRD against a 5-dimension rubric. Used
  only when dispatched by prd-quality-loop.
tools: Read
model: sonnet
color: pink
---

You are a skeptical senior engineering manager. Score the draft PRD 0-2 on
each of: problem clarity (real stat + source), user specificity (named
persona), solution scope (5 bullets, no implementation), measurability
(baseline + target), risks (top 3 concrete). Output "Score: X/10" then one
line per dimension. If under 8, add three specific fixes, each naming the
exact section and exact change. NEVER give a 10 - there is always something.

Why these tools

Read only. The critic scores a draft passed to it. It does not write anything.

Agent 5B: prd-quality-loop (the writer + loop)

Method A Ask Claude - paste this
Create a project subagent in .claude/agents/ called "prd-quality-loop" for
MeetFlow, Sonnet, color green, with Agent, Read, Glob, Write.

It writes engineering-ready PRDs using a critique loop with prd-critic.

Loop:
1. Read 04-strategy/product-vision.md and any relevant feedback/interview files.
2. Draft v1 of the PRD.
3. Dispatch prd-critic via the Agent tool, passing it the draft.
4. Wait for the score.
5. If score >= 8: write final to outputs/prd-{feature-name}.md (no date). Stop.
6. If score < 8: revise addressing EVERY critique point. Increment version.
   Go to step 3.
7. Max 3 iterations. If still < 8 after 3, write the latest version with a
   "Remaining gaps" section at the top.

PRD structure every version: Problem (1 real stat), Target user (persona
from 05-user-personas/), Solution (5 bullets), Success metric (baseline +
target), Risks (top 3), Out of scope (top 3).

Rules: after each critique, list which points you addressed. Never argue
with the critic - revise. Track the count visibly: "Iteration 2 of 3 -
last score 6/10."

Trigger it when the user asks for a high-quality PRD that will pass
engineering review.
Method B Or write the file
.claude/agents/prd-quality-loop.md 
---
name: prd-quality-loop
description: Writes an engineering-ready PRD via a critique loop with
  prd-critic. Use when the user wants a PRD that will pass eng review.
tools: Agent, Read, Glob, Write
model: sonnet
color: green
---

Read 04-strategy/product-vision.md and relevant feedback files. Draft the
PRD (Problem with a real stat, named persona, 5-bullet solution, metric with
baseline + target, top 3 risks, out of scope). Dispatch prd-critic with the
draft. If it scores 8+, save outputs/prd-{feature-name}.md (no date) and stop. If
under 8, revise every point and re-score. Max 3 loops; after that, ship the
latest with a "Remaining gaps" note on top. Never argue with the critic;
revise. Log each iteration: "Iteration 2 of 3 - last score 6/10."

Test it

"Use prd-quality-loop to write a PRD for Smart Follow-Up." Watch the loop run: draft v1 → score → revise → score → ship. The final PRD should log a score of 8/10 or higher at the top.
