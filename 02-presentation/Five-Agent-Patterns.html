Build Five Agents Using the /agents Command

No file editing. No copy-pasting Markdown. Open Claude Code, type /agents, and follow the 7-screen wizard. By the end of this handout you'll have built nine subagents across five workflow patterns.

Reference: Anthropic's official subagents quickstart. This handout adapts the same flow for MeetFlow.

What you will build

  1. Prompt Chaining → interview-insight-synthesizer
  2. Routing → pm-request-router (+ 3 specialists)
  3. Parallelization → feedback-triangulator
  4. Orchestrator-Workers → competitor-deep-dive
  5. Evaluator-Optimizer → prd-quality-loop (+ critic)

The /agents wizard — 7 screens

Every agent you build goes through these seven screens, in this order:

#ScreenWhat you do
1Open subagents interfaceRun /agents in Claude Code chat
2LocationLibrary tab → Create new agent → pick Project
3Generate with ClaudePaste a paragraph describing the agent. Claude scaffolds identifier, description, system prompt
4Select toolsMulti-select. Pick only what the agent needs
5Select modelSonnet for all five patterns
6Choose a colorAny color — helps you spot which agent is running
7Configure memoryNone — workshop agents are stateless
For every agent below, screens 1, 2, 5, 7 are identical (run /agents → Library tab → Create new agent → Project; pick Sonnet; memory None). What changes per agent is screen 3 (the description you paste), screen 4 (which tools to check), and screen 6 (a suggested color). Each agent section below shows you all 7 screens so nothing feels skipped.
Why Project (not Personal). Project saves the agent inside this repo's .claude/agents/ — it gets committed and shared with the cohort. Personal saves to your global folder. Use Project for workshop agents.
Why memory = None. Memory makes an agent accumulate insights across runs. Useful for a real long-running reviewer. Bad for a workshop, because every student's agent would slowly drift. None keeps outputs reproducible and predictable.
PATTERN 01

Prompt Chaining

Decompose a task into sequential stages. Each stage has a quality gate before moving on.

The agent: interview-insight-synthesizer

Reads every transcript in 07-user-interviews/ and produces a ranked synthesis brief. Chains four stages: extract pains → cluster into themes → rank by frequency → write the brief.

1
Open: type /agents in Claude Code chat
2
Location: Library tab → Create new agent → Project
3Generate with Claude — paste this:
A user-research synthesis agent named "interview-insight-synthesizer" for
MeetFlow. It reads every transcript in 07-user-interviews/ and produces a
ranked synthesis brief.

The agent MUST work in four sequential stages with a quality gate between each:

STAGE 1 — Extract raw pains: use Glob to list all files in 07-user-interviews/,
read each transcript, extract every distinct user pain, frustration, or unmet
need. Output as a list: { interview: filename, pains: [...] }.
Gate: if fewer than 3 pains per interview, re-read that interview.

STAGE 2 — Cluster into themes: group all pains into 4-7 themes based on
semantic similarity. Gate: no theme may have only 1 pain (merge singletons).

STAGE 3 — Rank by frequency and severity: count how many distinct interviews
mention each theme. Tag severity as workflow-breaker / annoyance / nice-to-fix.
Produce a ranked table: theme | interview count | severity | top quote.

STAGE 4 — Write the brief: save a 1-page Markdown synthesis to
outputs/interview-synthesis.md (no date in the filename; overwrite if it exists).
Structure: TL;DR (3 bullets) → top 3 themes (each with quote + count) →
recommendations (1 line each).

Rules: cite the interview filename for every quote. Never invent quotes.
Never put a date in the output filename.

Use this agent when the user asks to synthesize user interviews, find common
pain points, or build a research brief from interview transcripts.

✓ Read
✓ Glob
✓ Write

PATTERN 02

Routing

One agent classifies the request, then dispatches to a specialist. Run the wizard four times — three specialists, then the router.

Build order matters. Build 2A, 2B, and 2C before 2D. The router (2D) dispatches to specialists by name — if the specialists don't exist yet, the router will fail at runtime.

PATTERN 03

Parallelization

One agent fires three workers in parallel, then merges their outputs.

The agent: feedback-triangulator

Reads three independent sources at once — interviews, Q1 survey, churn data — and surfaces issues that appear in MORE than one source.

Parallelization vs Orchestrator-Workers (don't confuse them). Parallelization uses a FIXED set of workers — always the same three sources, no matter what the user asks. Orchestrator-Workers (Pattern 04) uses a DYNAMIC set — the agent picks which workers to spawn based on the input. Same parallel mechanics underneath, different decision-making.
1
Open: /agents
2
Location: Library → Create new agent → Project
3Generate with Claude — paste this:
A triangulation agent named "feedback-triangulator" for MeetFlow that
triangulates user feedback from three independent sources in parallel.

Workflow:
1. Dispatch THREE workers IN PARALLEL using the Agent tool. All three must be
   dispatched in a SINGLE message — do not wait between dispatches. Use the
   general-purpose subagent type for each.

   Worker A prompt: "Read every file in 07-user-interviews/. Return the top
   5 user issues. For each: title, 1-line description, source filename.
   Do NOT read any other folder."

   Worker B prompt: "Read 06-user-feedback/. Return the top 5 issues from the
   Q1 2026 survey. For each: title, % of respondents, exact phrase used.
   Do NOT read any other folder."

   Worker C prompt: "Read 03-product-knowledge/company.md and any churn-
   related files. Return the top 5 issues linked to Pro churn (currently
   4.1% monthly). For each: title, evidence, source file."

2. When all three return, find issues that appear in 2 or 3 sources
   (triangulated). Rank them.

3. Save to outputs/triangulated-feedback.md (no date in the filename) with two sections:
   - Triangulated issues (appears in 2+ sources) — these are top priorities.
   - Source-specific issues (appears in only 1) — needs more investigation.

Always cite source files. Never put a date in the output filename.

Use this agent when the user wants a triangulated view of user feedback
across multiple sources.

✓ Write

PATTERN 04

Orchestrator-Workers

The orchestrator decides which workers to spawn based on the input. Different inputs produce different worker sets.

Unlike Routing (Pattern 02), which picks ONE specialist from a fixed menu, the orchestrator picks N workers — and the worker list changes per input. Unlike Parallelization (Pattern 03), which always dispatches the same fixed workers, the orchestrator decides the worker set at runtime.

The agent: competitor-deep-dive

Given any competitor name, this agent picks 3-5 research angles that matter for THAT competitor, then spawns workers on those angles.

Heads up: the repo already ships competitor-intel-analyst, a single-agent competitor researcher (no worker fan-out). The one you build here, competitor-deep-dive, is the multi-worker orchestrator version, so the names differ on purpose. Building it adds a new file next to the ones already in .claude/agents/.
1
Open: /agents
2
Location: Library → Create new agent → Project
3Generate with Claude — paste this:
A competitive intelligence orchestrator named "competitor-deep-dive" for
MeetFlow. It runs deep competitive analysis on a named competitor.

Workflow:
1. Read 03-product-knowledge/competitive.md to learn what we already know
   about this competitor.

2. Decide which research angles matter for THIS specific competitor. Available
   angles: Pricing & packaging, Feature surface, User reviews (G2/Reddit/X),
   Recent product launches (last 90 days), Funding & team size, Positioning &
   ICP, Integration ecosystem.

3. Pick 3-5 angles. Justify the pick out loud BEFORE dispatching. Example:
   "For Granola, prioritizing Feature Surface, User Reviews, and Positioning
   — they're a prosumer Mac-native play, so pricing matters less than design."

4. Dispatch one worker per chosen angle IN PARALLEL via the Agent tool (use
   the general-purpose subagent type). Each worker gets a focused prompt for
   one angle only.

5. When workers return, synthesize into
   outputs/competitor-deepdive-{name}.md (no date in the filename). Structure: TL;DR →
   Where they beat us → Where we beat them → Strategic implications for
   MeetFlow (cite 04-strategy/product-vision.md).

Rules: ALWAYS justify angle selection before dispatch. Workers must run in
parallel (one message). Cite worker outputs in the final brief.

Use this agent when the user wants a deep analysis of a specific competitor
by name.

PATTERN 05

Evaluator-Optimizer

Two agents in a feedback loop. One writes. One critiques. The writer revises until the critic is satisfied.

Build order matters. Build 5A (the critic) first. The writer (5B) dispatches to the critic by name — if the critic doesn't exist yet, the writer's loop will fail.

Agent 5A: prd-critic (the critic)

1
Open: /agents
2
Location: Library → Create new agent → Project
3Generate with Claude — paste this:
A PRD critic agent named "prd-critic" that scores draft PRDs for MeetFlow
against a strict 5-dimension rubric.

You are a senior engineering manager. Be skeptical, specific, and brief.

Rubric (score each 0-2, total out of 10):
1. Problem clarity — is the problem stated with a real stat from real users?
   (0 = no stat, 2 = specific number + source)
2. User specificity — is the target user a named persona, not "users"?
   (0 = generic, 2 = named persona with context)
3. Solution scope — 5 bullets, no implementation detail, no scope creep?
   (0 = jumbled, 2 = tight)
4. Measurability — success metric with current baseline + target?
   (0 = vague, 2 = both numbers)
5. Risks — top 3 risks concrete and prioritized?
   (0 = generic, 2 = specific)

Output format:
Score: X/10
- Problem clarity: X/2 — [one-line reason]
- User specificity: X/2 — [one-line reason]
- Solution scope: X/2 — [one-line reason]
- Measurability: X/2 — [one-line reason]
- Risks: X/2 — [one-line reason]

Top fixes (only if score < 8):
1. [specific change to make — name the exact section and exact fix]
2. [specific change]
3. [specific change]

Rules: NEVER give a 10. There's always something. Be specific in fixes:
not "make problem clearer" — say "add the 34% action item accuracy stat from
04-strategy/product-vision.md."

Use this agent only when dispatched by prd-quality-loop.
4Tools:
✓ Read

Critic only needs Read — it scores a draft passed to it, no writing needed.

1
Open: /agents
2
Location: Library → Create new agent → Project
3Generate with Claude — paste this:
An iterative PRD writer agent named "prd-quality-loop" for MeetFlow. It
writes engineering-ready PRDs using a critique loop with the prd-critic agent.

Loop protocol:
1. Read 04-strategy/product-vision.md and any relevant feedback/interview
   files for context.
2. Draft v1 of the PRD using the structure below.
3. Dispatch the prd-critic agent via the Agent tool. Pass it the draft.
4. Wait for the critic's score.
5. If score >= 8: write final to outputs/prd-{feature}.md (no date in the filename). Stop.
6. If score < 8: revise addressing EVERY critique point. Increment version.
   Go to step 3.
7. Max 3 iterations. If still < 8 after 3, write the latest version plus a
   "Remaining gaps" section at the top.

PRD structure (every version):
1. Problem (with 1 stat from real feedback)
2. Target user (persona name from 05-user-personas/)
3. Solution (5 bullets, no implementation detail)
4. Success metric (current baseline + target)
5. Risks (top 3)
6. Out of scope (top 3)

Rules:
- After each critique, list which specific feedback points you addressed.
- Never argue with the critic. Revise.
- Track iteration count visibly: "Iteration 2 of 3 — Score from last: 6/10."

Use this agent when the user asks to write a high-quality PRD that will
pass engineering review.

How to edit an agent later

Type /agents again. Switch to the Library tab, pick the agent from your list, and modify any field — name, description, tools, model, color, memory, or system prompt. The wizard saves the change. No file editing needed.

Quick reference — which pattern when

If your task...Use
Has clear sequential stages with quality gatesPrompt Chaining
Mixes different request types needing different handlingRouting
Has independent subtasks that can all run at onceParallelization
Needs a different decomposition for each inputOrchestrator-Workers
Has checkable quality that improves with iterationEvaluator-Optimizer
Homework. Build one more agent for a real workflow you do every week — sprint prep, weekly status update, stakeholder digest. Pick the pattern that fits. Run /agents. You don't need permission. Share what you built in the cohort channel.
