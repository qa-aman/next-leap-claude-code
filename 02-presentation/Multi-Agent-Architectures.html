Teaching reference for PMs

Multi-Agent Systems for PMs

A simple guide to using multiple AI agents together in Claude Code. Every idea comes with a PM example, plus a step-by-step practical you can run live in this repo (next-leap-claude-code) during the session.

How to use this doc
Blue boxes = the PM-shaped example (the "why this matters" story). Green boxes = "Try it now in this repo" — copy the prompt, paste into Claude Code, and run the workflow live. Every practical uses files that already exist in this repo.

00 PM glossary

Plain-English definitions before we go any deeper. Skip if these terms are already familiar.

Agent
A Claude session doing work for you. It can read files, search the web, write things, and use tools. Think of it as a smart helper with a toolbox.
Context window
The agent's working memory for this conversation. Everything it has read so far sits in there. When it gets too full, the agent starts forgetting things.
Tool call
When the agent uses something outside the chat: reading a file, searching the web, opening a Jira ticket. Each one adds to its memory.
Subagent
A helper that the main agent sends off to do a side task. The helper has its own memory, finishes the task, and hands back a short summary. Like asking a junior PM to do some digging and send you the headline.
Lead (orchestrator)
The main agent that plans the work, splits it up, hands pieces to helpers, and combines the results. Think tech lead or program manager.
Agent team
A group of agents working together with a shared to-do list. Unlike subagents, teammates can talk to each other. Like a small cross-functional pod with a lead.
Worktree
A separate copy of your project files, so two agents can edit without overwriting each other. Like giving two people their own branch of the code.
MCP
The plug system that lets Claude connect to outside tools like Slack, Drive, Figma, and Jira. If you've added an integration, you've used MCP.

00 Which pattern do I use?

A quick cheat sheet before we dive in. Use this to pick the right tool for any task.

I want a side task done without cluttering my main chat
→
Use a subagent
I want to compare 3+ options or look at 3+ angles at once
→
Use an agent team (3 to 5 helpers)
I want to start a long task and check back later
→
Use agent view (runs in the background)
My agents will edit the same files at the same time
→
Use worktrees so they don't clash
It's a small linear task with no real parallel pieces
→
Use a single session — skip multi-agent

01 Why use multiple agents

Why one agent isn't always enough, and what changes when you use a few of them together.

A single Claude session has one memory. As your task grows, that memory has to hold the goal, the plan, the half-done work, and everything the agent has read so far. Quality starts dropping well before the memory is full. The agent forgets early details, repeats itself, and misses things.

Multi-agent fixes this by giving each agent its own fresh memory. Each helper looks at a different slice of the problem, and only sends back a short summary. The main agent never has to read the raw stuff. Its memory stays clean and focused on the final answer.

Headline number
Anthropic's multi-agent research system (one lead + several helpers) outperformed a single agent by 90.2% on their internal test. The win came from running agents in parallel and keeping the lead's memory clean — not from using a smarter model.
PM example

Summarizing user interviews into themes

If you load 50 transcripts into one Claude session, it loses track by around interview #20. The memory is full of raw text, so by the time you ask for themes, the agent has half-forgotten what was in interview #3. The output gets vague and you end up doing the work yourself.

Multi-agent fixes this. Send one helper per few interviews to pull out the themes. The lead only reads the short summaries (not the raw transcripts) and finds the patterns across them. You get a clean ranked list of themes without overloading anyone's memory.

When this approach wins

  • Open-ended research where you don't know in advance what you'll find.
  • Many things to look at side by side — multiple competitors, multiple hypotheses, multiple personas.
  • Tasks too big for one memory — long investigations, big folders of files, wide comparisons.
  • Work that's important enough to spend extra tokens on better quality.

What it costs

4×
More tokens than a normal chat for a single agent
15×
More tokens than a normal chat for a multi-agent run
80%
Of result quality on research tasks is explained by how many tokens you spend
~90%
Less time to finish when agents work in parallel instead of one by one

Multi-agent trades extra tokens for better answers. It isn't always the right tool. For coding tasks, for example, there usually aren't enough truly independent pieces to make it worth the cost.

02 The four ways to run agents in parallel

Claude Code gives you four options. Each one is a different way to split the work — and below each is a step-by-step practical you can run live in this repo.

Pattern What it does When to use it
Subagents Helpers inside one session. Each has its own memory and returns a summary. A side task would clutter your main chat with results, logs, or file contents you won't need again.
Agent viewpreview One screen to start and check on agents running in the background. Open with claude agents. You have a few separate tasks. You want to start them, glance at progress, and step in only if needed.
Agent teamsexperimental A group of agents with a shared to-do list. Teammates can message each other. A lead keeps them in sync. You want Claude to split a project into pieces, hand them out, and have the team coordinate.
Worktrees Separate copies of your project files, so two agents never edit the same file at the same time. You're running multiple agents that will all write to the codebase.

03 The manager-and-helpers setup

The setup that powers almost every serious multi-agent system. Engineers call it "orchestrator-worker" — but it's really just a manager handing out work.

Almost every multi-agent system uses the same shape: one lead (the manager) that plans and combines, and several helpers that do the actual digging. The lead never does the deep work itself. The helpers never see each other's raw notes.

┌──────────────────────────┐ │ Lead / Manager │ │ - Reads what you want │ │ - Splits the work │ │ - Sends out helpers │ │ - Combines the results │ └──────────┬───────────────┘ │ ┌─────────────────────┼─────────────────────┐ ▼ ▼ ▼ ┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐ │ Helper A │ │ Helper B │ │ Helper C │ │ own memory │ │ own memory │ │ own memory │ │ own tools │ │ own tools │ │ own tools │ │ returns: SUMMARY│ │ returns: SUMMARY│ │ returns: SUMMARY│ └─────────────────┘ └─────────────────┘ └─────────────────┘
PM example

Mapping the AI meeting assistant market

You ask the lead: "I need a competitive matrix of the top 5 AI meeting assistants — pricing, free-tier limits, top differentiators, and recent launches."

The lead splits the work: one helper per competitor — Notion AI, Granola, Fireflies, Otter, tl;dv. Each helper digs into one company's pricing page, blog, recent reviews, and returns a one-page summary.

The lead combines: turns the 5 summaries into a comparison table, flags the threats, and suggests 3 strategic moves. You never opened 5 competitor websites yourself.

  • A fresh matrix lands at outputs/competitive-scan-23-05-2026.md with source URLs.
  • The lead asks you to confirm the updates to competitive.md rather than silently overwriting.

Why this works better than one big agent

  1. Each helper keeps the lead's memory clean. Helpers burn their own memory on digging. The lead only sees the short summaries.
  2. Helpers run at the same time. The lead sends out 3–5 helpers at once. The task finishes much faster than doing it one step at a time.
  3. Each helper is focused. One narrow brief and one job. Quality goes up when the role is narrow.
  4. A bad helper doesn't ruin the whole run. If one helper gets stuck or returns junk, it doesn't pollute the lead's thinking.

Subagents vs. agent teams — one key difference

Subagents

  • Own memory. Results go back to whoever sent them.
  • Can't talk to each other.
  • The main agent runs the show.
  • Best for focused tasks where you only need the final answer.
  • Cheaper. The results get summarized into the main chat.

Agent teams

  • Own memory. Fully separate sessions.
  • Teammates can message each other directly.
  • Shared to-do list. The team coordinates itself.
  • Best for complex work that needs discussion.
  • More expensive. Each teammate is its own full Claude session.

04 Real example: Anthropic's research system

The system Anthropic actually shipped — and most of what we know about making multi-agent work in real use comes from here.

The flow

  1. The lead reads your question and plans an approach.
  2. It sends out helpers at the same time.
  3. Each helper searches, looks at results, and refines its search based on what it finds.
  4. Helpers send back short findings.
  5. The lead combines them. If something's missing, it sends out more helpers.
  6. A separate "citation" helper attaches a source link to every claim, so nothing is unsourced.
"Helpers keep things compact by running in parallel with their own memory, looking at different angles at the same time, and then handing the lead only the short version."
PM example

"Map the AI meeting assistant market in 60 minutes — with sources I can put in my PRD"

Same setup as Anthropic's. The lead splits the task into 5 helpers, one per competitor. Each helper looks at pricing pages, recent blog posts, and reviews. A citation helper attaches a source URL to every claim. So when your VP asks "where did this number come from", you have the link in your PRD — not "Claude said so."

05 8 things to keep in mind when prompting

The hard-earned lessons from real multi-agent runs in production. Each one comes with a PM example, plus a single combined practical at the end you can run live.

  1. Look at what the agent is actually doing. When something goes wrong, replay the trace step by step. Don't guess. The agent usually did exactly what you asked, just not what you meant.
    PM lens

    Same rule as user research: don't guess what the user is doing — go watch. When a multi-agent run gives you a weird output, look at the trace before you "fix the prompt."

  2. Be specific when you delegate. "Research semiconductors" is too vague. Tell the helper what you want, what the output should look like, which tools to use, and where its job ends. Without this, helpers either repeat each other or leave gaps.
    PM lens — vague vs. specific

    Vague: "Research Notion AI."

    Specific: "Goal: find Notion AI's pricing tiers, free-tier limits, and the 3 things they push the most on their homepage. Output: a one-page summary with bullets and source URLs. Tools: web search only. Don't cover: enterprise SKUs or partner programs."

  3. Match the effort to the question. Simple fact = one agent. Comparison = a few helpers. Big research project = many helpers with clear roles. Don't spin up a team for a one-line question.
    PM lens

    "What's our current Pro churn rate?" = one session, look it up. "Why are Pro users churning?" = one helper for tickets, one for survey responses, one for usage data, lead combines.

  4. Be careful with tools. The tools you give an agent shape what it does. Tell it which tool to use for which job, and which to avoid.
    PM lens

    If your agent has Slack, Jira, and Confluence connected, tell it: "Use Jira for tickets, Confluence for specs, and Slack only for recent decisions." Otherwise it might paste your spec into a Slack DM.

  5. Let the agent improve itself. Claude can review its own runs and suggest better prompts. Anthropic built an agent that rewrites tool descriptions — and got 40% faster runs on the next tasks because of it.
    PM lens

    After a multi-agent run, ask Claude: "Look at the trace. Which step wasted the most tokens, and how would you rewrite the prompt to fix it?"

  6. Start wide, then go narrow. Agents tend to jump straight to specific questions. Force them to start with "what's broadly known about this?" before going deep.
    PM lens

    Same idea as good discovery: start with "what is the user trying to do" before "should we add this button."

  7. Ask the agent to think before doing. Get the lead to write a plan before sending out helpers. This catches bad plans early.
    PM lens

    Tell the lead: "First, write me the plan for how you'll split this across 5 helpers. I'll approve before you send them out."

  8. Make sure things run in parallel, not one at a time. The lead should send out 3–5 helpers at the same time. Each helper should call its tools at the same time too.
    PM lens

    If your agent is doing research one company at a time and you're waiting 20 minutes, ask directly: "Send out 5 helpers at the same time, one per competitor."

06 How to manage agent memory

Prompting is what you say at the start. Memory management is everything you do across a long task to keep the agent sharp.

Every token in the lead's memory is space taken away from the final answer. On long runs, what you choose not to load matters as much as what you do load.

PM example

The 90-minute roadmap session that forgot Q2

You spend the first 30 minutes loading Q2 themes into Claude — user research, OKRs, competitive pressure. Then you spend an hour going deep on Q3. By minute 90, the agent has "forgotten" Q2 because Q3 details have pushed them out of effective memory.

Fix: save the Q2 themes to a file (q2-themes.md) and tell the agent: "Read this file when you need Q2 context." Now Q2 is one tool call away instead of crowding the conversation.

What success looks like
  • Helpers fetch files on demand instead of having them pre-loaded.
  • The lead's chat stays short — only summaries come back.
  • The final memo cites which source each signal came from.

Seven simple rules

  1. Memory is a limited resource. Agents get worse at remembering as their memory fills up — well before you hit the official limit.
  2. Use the smallest amount of useful info, not the most. Be ruthless about what you include.
  3. Don't be too rigid or too vague. Give strong guidance but leave room to adapt.
  4. Keep tools focused. If you can't tell which tool applies, the agent can't either.
  5. Load info only when needed. Give the agent paths or links, let it fetch on demand.
  6. Clear old tool results. Once a tool output has been used and summarized, the raw result is just noise.
  7. Use helpers for complex tasks. Send the deep work to helpers with their own memory. Bring back only the short version.
The big idea
Multi-agent research is really about squeezing big info into small summaries. Helpers do this better because each one has a clean slate and focuses on one slice.

07 Agent teams in Claude Code experimental

The official Claude Code feature for running coordinated agent teams. (Practical for setup + first run is in section 2C above — this section covers the deeper concepts.)

What's inside an agent team

PieceWhat it does
Team leadThe main session. Creates the team, sends out teammates, keeps things on track.
TeammatesSeparate Claude sessions, each working on their assigned piece.
To-do listA shared list of tasks teammates can pick up. Locked so two people can't grab the same task.
MailboxHow teammates send messages to each other directly.

Two ways to view the team

In your main terminal (default)

All teammates run in your terminal. Use Shift + ↓ to flip through them and message them directly. Works anywhere — no setup.

Split into separate panes

Each teammate gets its own pane (needs tmux or iTerm2). You can see everyone's output at once. Best when you want to watch closely.

PM example — investigation with competing theories

Find why conversion dropped 30% last month

A single agent picks the first plausible theory and stops. An agent team with 4 competing theories debates, tries to disprove each other, and the surviving one is much more likely to be the real answer.

Investigation: why is Pro churn at 4.1% / month? (competing theories)

Prerequisites

  • Claude Code in the repo
  • Files exist at 06-user-feedback/, 05-user-personas/, 03-product-knowledge/
Steps
  1. Start Claude Code: claude
  2. Paste the prompt below. The lead spawns 4 teammates, each owning one theory.
  3. Use Shift + ↓ to flip through teammates and watch the debate.
  4. The lead writes up the surviving theory and the alternatives.
The prompt
MeetFlow Pro churn is 4.1% monthly (from company.md). I want
to find the most likely cause.

Create an agent team. Spawn 4 teammates, each owning one theory:
1. Theory A: Action item accuracy (66%) is eroding trust
2. Theory B: Notion AI bundling is pulling users away
3. Theory C: Onboarding is failing to demonstrate value
4. Theory D: It's a measurement / cohort issue, not a real change

Each teammate investigates its own theory using:
- 06-user-feedback/ for verbatim signal
- 05-user-personas/ for who's churning
- 03-product-knowledge/ for context on product and competitive

Then have them debate openly. Each must try to disprove the
others' theories before the lead picks a verdict.

The lead writes:
- The surviving theory with evidence
- The 2 next most likely alternatives
- 3 next investigations to confirm

Save to outputs/pro-churn-rca-23-05-2026.md.
What success looks like
  • You see real disagreement in the teammate output — not 4 agreeable summaries.
  • The verdict cites specific evidence (which file, which quote).
  • The "next investigations" list is something you could actually run next week.

Best practices

  • Start with 3–5 teammates. More than that, the coordination cost gets bigger than the benefit.
  • Give them enough context. Teammates read CLAUDE.md automatically, but they don't see the lead's chat history. Be clear in the spawn prompt.
  • Don't let them fight over files. Give each teammate a different file or section.
  • Start with research or review, not building. Lower risk, clearer success criteria, less coordination needed.
  • Make sure the lead waits. If the lead starts doing the work itself, tell it to wait.

08 When NOT to use multiple agents

Multi-agent is a power tool. Use it for the right jobs.

Skip multi-agent when

  • The task is step-by-step — step 2 needs the answer from step 1.
  • Every agent would need to see the same big shared context.
  • The work touches the same files heavily (overwrites).
  • It's a small task. The coordination cost isn't worth it.
  • You can't name clear, separate roles for each agent.
  • You're in a focused one-on-one chat that benefits from continuity.

Use multi-agent when

  • You want to compare many things or explore many angles at once.
  • Each helper can do something useful on its own.
  • The work would otherwise overflow one agent's memory.
  • You want to fight first-idea bias by making the agents debate.
  • Quality matters more than token cost for this task.
  • You can wait for the combining step at the end.
PM tasks — which is which
PM taskUse this
Write one PRD from a briefSingle session (step-by-step, shared context)
Summarize 10 sprint retros into themesMulti-agent (parallel, many inputs)
Prep for one user interviewSingle session (focused conversation)
Quarterly competitive scan of 5 companiesMulti-agent (parallel research, sources)
Draft a Jira ticket from a specSingle session (small, sequential)
Stress-test a roadmap with 4 stakeholder lensesAgent team (debate)
Reply to a Slack threadSingle session (small, conversational)
Find why conversion dropped 30%Agent team (competing theories)
Be honest about cost
Multi-agent costs about 15× the tokens of a normal chat. Before you reach for it, ask: would a single Claude session get me 80% of the way there? Often yes.

09 Common PM mistakes

The 4 mistakes PMs make on their first runs. Avoid these and you'll save hours.

Mistake 1: Using a team for a task that doesn't need one

"I'll use an agent team to write this user story." Don't. That's one focused task, one session. Multi-agent is for breadth — many angles, many inputs, many things to compare. If you can't list 3+ truly independent angles, stick with one session.

Mistake 2: Vague briefs for helpers

"Research the competitor" gets you garbage back. "Find pricing tiers, free-tier limits, and the 3 most-promoted differentiators; output a one-page summary with source URLs" gets you something usable.

Mistake 3: Letting agents fight over files

You ask the team to "update the PRD" and three teammates start editing the same file. Last one wins. Either give each teammate a different section, or use worktrees so each has its own copy.

Mistake 4: Only watching the lead

You watch the lead's screen the whole time. Meanwhile a helper is stuck in a loop, burning tokens. Use Shift + ↓ to flip through teammates during the run. Your first 2–3 multi-agent runs should be 100% supervised.

The honest truth
Your first multi-agent run will probably be worse than a single session. The skill is in the briefing and the supervision, not in the tool. Plan to throw away your first 2–3 runs as learning fees.

10 PM playbook — 5 ready-to-run prompts

Copy any of these straight into Claude Code. Each is a full multi-agent workflow for a common PM job. Just swap in your details where you see [brackets].

1. User interview synthesis

I have [N] user interview transcripts in [folder/path].

Send out one helper per transcript. Each helper should:
- Find the user's top 3 pain points
- Tag each one with: how often it comes up (one-off vs recurring),
  how serious it is (annoying / blocking / show-stopper), and
  which product area it touches
- Quote the exact line that supports the tag

After all helpers report back, combine across them:
- Top 5 themes by how often they appear
- Top 5 themes by how serious they are
- 3 themes that surprised you (rare but very serious)

Output: outputs/interview-synthesis-[DATE].md

2. Competitive landscape scan

I need a competitive scan of [list of N competitors] for
[my product / feature area].

Send out one helper per competitor. Each one should:
- Find the current pricing tiers and free-tier limits
- Find the 3 things they push the most on their homepage
- Find any product launches or major changes in the last 90 days
- Cite a source URL for every claim

After all helpers report back, combine into:
- A comparison table
- 3 strategic threats ranked by urgency
- 3 strategic openings we should consider

Output: outputs/competitive-scan-[DATE].md

3. Adversarial PRD review

Create an agent team to pressure-test the PRD at [path/to/prd.md].

Spawn three teammates:
- One playing a skeptical Engineering Lead. Push back on effort
  estimates, hidden complexity, scalability risks.
- One playing Design. Push back on UX assumptions, edge cases,
  accessibility.
- One playing Sales / GTM. Push back on whether this is sellable,
  whether the value is clear, and whether enterprise will care.

Have them debate. After 2 rounds, write up:
- Top 5 issues I should fix before the eng review
- 3 questions I should have answers to before the meeting
- 1 thing in the PRD that's actually strong and should stay

Output: outputs/prd-review-[DATE].md

4. Finding the cause of a metric drop

Our [metric] changed by [X%] over the last [time period].
The data is at:
- [path to usage data]
- [path to tickets / surveys]
- [path to release notes / changelog]

Spawn 4 agent teammates, each owning one theory:
1. A recent product change caused it
2. A pricing or packaging change caused it
3. A competitor move caused it
4. It's a data / instrumentation issue, not a real change

Each teammate investigates its own theory using the data above.
Then they debate to try to disprove each other.

Combine the results:
- The surviving theory with evidence
- The two most likely alternatives
- 3 next investigations to confirm

Output: outputs/rca-[metric]-[DATE].md

5. Roadmap stress-test by stakeholder

Stress-test our [Q3 2026] roadmap at [path/to/roadmap.md].

Spawn 4 agent teammates, each playing a different stakeholder:
- CEO: Does this move the company-level metrics?
- Engineering Lead: Is the scope realistic for the team size
  and known dependencies?
- Top Customer (use persona at [path/to/persona.md]): Does this
  solve their actual problems?
- Competitor analyst (use competitive scan at [path]): Does this
  keep us competitive? What gap stays unfilled?

Have them debate the roadmap. Combine the results:
- Top 3 things to add
- Top 3 things to cut or push to later
- 1 contrarian bet worth considering

Output: outputs/roadmap-stress-test-[DATE].md

11 Your first run, step by step

The shortest path from "I read about this" to "I'm using it."

  1. Pick a low-stakes task. Summarize last week's user interviews. Don't start with a quarterly OKR debate.
  2. Try subagents first. Simpler, cheaper, built in. Run the practical in section 2A.
  3. Run the interview synthesis practical (section 1) as your first real run. It's a real PM workflow with very low risk.
  4. Then try agent teams with the adversarial PRD review (section 2C).
  5. Watch one full run end-to-end using Shift + ↓. You'll learn more in 10 minutes of watching than 2 hours of reading.
  6. Add hooks for safety once you run teams regularly.
  7. Track what works. Note token cost and quality on your first 3–5 runs.
